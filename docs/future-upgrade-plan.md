⏺ 좋은 질문입니다. 현재 아키텍처를 분석해보면, 분산 실행의 핵심 병목은 3가지입니다.                                                                                            
                                                                                                                                                                               
  현재 한계점                                                                                                                     
                                                                                                                                                                               
  ┌──────────────┬───────────────────────────┬────────────────────────────┐                                                                                                    
  │   구성요소   │           현재            │            문제            │                                                                                                    
  ├──────────────┼───────────────────────────┼────────────────────────────┤                                                                                                    
  │ 시그널 파일  │ 로컬 /tmp/claude-signals/ │ 다른 머신에서 보이지 않음  │
  ├──────────────┼───────────────────────────┼────────────────────────────┤
  │ Git worktree │ 로컬 .claude/worktrees/   │ 다른 머신에 없음           │                                                                                                    
  ├──────────────┼───────────────────────────┼────────────────────────────┤                                                                                                    
  │ Worker 관리  │ tmux send-keys (로컬 IPC) │ 원격 머신의 tmux 제어 불가 │                                                                                                    
  └──────────────┴───────────────────────────┴────────────────────────────┘                                                                                                    
                                                              
  분산 실행에 필요한 것들                                                                                                                                                      
                                               
  1. WP 단위 분배 (가장 자연스러운 경계)                                                                                                                                       
                                               
  각 머신이 서로 다른 WP를 맡는 구조:                                                                                                                                          
  Machine A: /dev-team WP-01 WP-02   (팀리더 A)
  Machine B: /dev-team WP-03 WP-04   (팀리더 B)                                                                                                                                
                                                                                                                                                                               
  WP 내부는 기존 로컬 tmux 방식 그대로 동작. 머신 간 통신은 WP 간 의존성 해소에만 필요.
                                                                                                                                                                               
  2. 공유 시그널 레이어 (핵심)                                                                                                                                                 
                                                                                                                                                                               
  로컬 파일 시그널을 원격으로 확장해야 합니다. 선택지:                                                                                                                         
                                                              
  ┌─────────────────────────────┬─────────────────────────────────┬─────────────────────────────────────┐                                                                      
  │            방식             │              장점               │                단점                 │
  ├─────────────────────────────┼─────────────────────────────────┼─────────────────────────────────────┤
  │ Git remote (push/poll)      │ 추가 인프라 없음, 이미 git 사용 │ 폴링 지연 (10~30초), push 충돌 가능 │
  ├─────────────────────────────┼─────────────────────────────────┼─────────────────────────────────────┤
  │ 공유 파일시스템 (NFS/SSHFS) │ 코드 변경 최소                  │ 네트워크 마운트 설정 필요           │                                                                      
  ├─────────────────────────────┼─────────────────────────────────┼─────────────────────────────────────┤                                                                      
  │ Redis/Valkey                │ 실시간 pub/sub, 빠름            │ 외부 서비스 의존                    │                                                                      
  ├─────────────────────────────┼─────────────────────────────────┼─────────────────────────────────────┤                                                                      
  │ SSH + 파일 동기화           │ 간단, 인프라 최소               │ 양방향 동기화 복잡                  │
  └─────────────────────────────┴─────────────────────────────────┴─────────────────────────────────────┘                                                                      
                                               
  가장 실용적인 안: Git remote를 시그널 버스로 쓰는 것. 이미 git repo가 있고, 각 머신이 브랜치를 push하므로.                                                                   
                                               
  3. 구체적으로 바뀌어야 할 것들                                                                                                                                               
                                               
  A. 시그널 프로토콜 확장                                                                                                                                                      
  현재:  touch /tmp/claude-signals/TSK-01-01.done  (로컬)
  변경:  git push + 원격 시그널 브랜치  또는  ssh remote "touch ..."                                                                                                           
                                                                                                                                                                               
  B. 분산 코디네이터 (새로 필요)                                                                                                                                               
  - 어떤 머신이 어떤 WP를 담당하는지 등록                                                                                                                                      
  - Cross-WP 의존성 시 원격 시그널 대기                                                                                                                                        
  - 전체 완료 판단 및 최종 머지 조율                                                                                                                                           
                                                                                                                                                                               
  C. 머지 전략 변경                                                                                                                                                            
  현재:  팀리더가 로컬 worktree를 main에 직접 merge                                                                                                                            
  변경:  각 머신이 WP 브랜치를 remote에 push                                                                                                                                   
         → 하나의 "마스터 머신"이 순차 pull + merge                                                                                                                            
         → 또는 PR 기반 merge                                                                                                                                                  
                                                                                                                                                                               
  D. WBS 잠금 (Lock)                                                                                                                                                           
  - 두 머신이 동시에 wbs.md의 같은 Task status를 변경하면 충돌                                                                                                                 
  - Task 할당 시 원격 lock 필요 (git tag, 별도 lock 파일, 또는 WP 단위 분리로 회피)                                                                                            
                                                                                                                                                                               
  4. 최소 구현 스펙                                                                                                                                                            
                                                              
  가장 단순하게 만든다면:                                                                                                                                                      
                                               
  [전제]                                                                                                                                                                       
  - GitHub/GitLab 등 공유 remote가 있음        
  - 각 머신에 동일 repo clone + claude 설치                                                                                                                                    
                                                                                                                                                                               
  [프로토콜]                                                                                                                                                                   
  1. 마스터 머신이 WP 분배표를 remote의 signal 브랜치에 push                                                                                                                   
  2. 각 머신이 자기 WP를 pull → 로컬에서 /dev-team 실행                                                                                                                        
  3. WP 완료 시 브랜치를 remote에 push + signal 브랜치에 완료 표시                                                                                                             
  4. Cross-WP 의존: signal 브랜치를 poll하여 의존 Task 완료 확인                                                                                                               
  5. 마스터 머신이 모든 WP 브랜치를 순차 merge                                                                                                                                 
                                                                                                                                                                               
  새로 만들어야 할 컴포넌트:                                                                                                                                                   
  - signal-helper.sh → signal-remote.sh (git push/pull 기반 원격 시그널)
  - dev-team SKILL.md에 --distributed 모드 추가                                                                                                                                
  - 분산 코디네이터 로직 (마스터 머신용)       
  - WP 할당 manifest를 remote에 게시하는 흐름
