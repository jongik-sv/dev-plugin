## 파일 계획

**경로 기준:** 모든 파일 경로는 **프로젝트 루트 기준**으로 작성한다. 모노레포에서 "타겟 앱"이 `apps/X`이면 경로에 `apps/X/` 접두어를 반드시 포함한다. 예: `apps/admin-web/src/pages/profile.tsx` (O), `src/pages/profile.tsx` (X). 단일 앱 프로젝트여도 동일하게 루트 기준으로 작성한다 (혼동 방지).

| 파일 경로 | 역할 | 신규/수정 |
|-----------|------|-----------|
| {타겟앱}/path/to/file | 설명 | 신규 |

> UI가 있는 Task(`fullstack`/`frontend`)는 이 표에 **라우터 설정 파일과 메뉴/네비게이션 파일**을 반드시 포함해야 한다 (아래 "진입점" 섹션 참조). 파일 계획에 없으면 dev-build가 구현 범위 밖으로 간주하여 배선을 누락한다.

## 진입점 (Entry Points)

**대상**: `domain`이 `fullstack` 또는 `frontend`인 Task (비-UI Task는 "N/A" 한 줄로 마무리).

WBS Task의 `entry-point` 필드를 그대로 옮기고, 실제 코드 레벨에서 "어디를 수정해야 하는지"를 구체화한다.

**작성 기준 — 각 항목은 코드 수정 시 즉시 특정 가능한 식별자까지 기입한다** (파일 경로만으로 부족하면 함수·배열·컴포넌트명을 함께 명시):

- **사용자 진입 경로**: 클릭할 메뉴/버튼명을 순서대로. 예: `로그인 → 좌측 사이드바 '설정' 클릭 → '프로필' 탭 클릭`
- **URL / 라우트**: 정확한 경로. 예: `/settings/profile`
- **수정할 라우터 파일**: 파일 경로(루트 기준) + 수정 위치. 예: `apps/web/src/routes.ts`의 `routes` 배열에 `/settings/profile` 엔트리 추가, `apps/web/src/app/settings/profile/page.tsx` 신규 생성 — 위 "파일 계획" 표에도 반드시 포함
- **수정할 메뉴·네비게이션 파일**: 파일 경로(루트 기준) + 수정할 변수/컴포넌트명. 예: `apps/web/src/components/Sidebar.tsx`의 `navItems` 배열 — 위 "파일 계획" 표에도 반드시 포함
- **연결 확인 방법**: E2E에서 검증할 클릭 시퀀스를 구체적으로. 예: `사이드바 '설정' 클릭 → '프로필' 탭 클릭 → URL이 /settings/profile로 변경됨`. URL 직접 입력(`page.goto`) 사용 금지 (dev-test reachability gate).

> **비-페이지 UI** (공통 컴포넌트 라이브러리, 디자인 시스템 등): "적용될 상위 페이지" 1개 이상을 기입하고, 해당 페이지의 E2E에서 해당 컴포넌트 렌더링을 검증한다.
